## 1. Introducción

### 1.1 Propósito
Este documento especifica los requisitos funcionales y no funcionales del sistema de gestión integral para la **Caja de Ahorro y Crédito Solidaria Familiar Pujota-Simbaña**, una organización de economía popular y solidaria ecuatoriana. El sistema automatiza la gestión de socios, créditos, inversiones, cobros, sesiones mensuales, multas, asistencias, notificaciones y portal de autogestión para socios.

### 1.2 Alcance
Sistema web MVC desarrollado en PHP 8.4 + MySQL 8.4, con arquitectura monolítica modular, frontend responsivo con Bootstrap 5.3, notificaciones en tiempo real vía Pusher, envío de correos SMTP con PHPMailer y autenticación de dos factores (2FA) por PIN.

### 1.3 Definiciones y abreviaturas
- **Socio:** Miembro de la caja de ahorro con derechos y obligaciones según el estatuto
- **Sesión mensual:** Reunión mensual ordinaria/extraordinaria donde se realizan cobros, check-in y operaciones financieras
- **Obligación:** Deuda generada al abrir sesión (cuota mensual, cuota crédito, multa)
- **RBAC:** Role-Based Access Control — control de acceso basado en roles
- **2FA:** Autenticación de dos factores vía PIN de 6 dígitos por correo
- **Endosable:** Rol que hereda TODOS los permisos del sistema
- **Excedentes:** Distribución de utilidades entre socios (cálculo anual)
- **SHA-256:** Hash criptográfico usado para integridad de historial financiero
- **UUID:** Identificador único universal (PK de todas las tablas)

### 2.4 Convenciones de Codificación
- **Texto de interfaz:** usar acentos y ñ (ej: `'Cédula'`, `'Inversión'`)
- **Identificadores de código:** sin acentos ni ñ (ej: `cedula`, `inversion`)
- **Nombres BD:** clean ASCII (ej: `caja_ahorro_pujota`)
- **UUIDs:** como PK en tablas transaccionales
- **CONCAT_WS:** para nombres completos
- **Historial:** solo INSERT, con SHA-256 para integridad
- **Archivos:** siempre por FileManager, nunca acceso directo
- **Form POST:** controllers redirigen con flash; JSON solo para AJAX
- **UTF-8 sin BOM:** en todos los archivos PHP

## 4. Requisitos No Funcionales

### 4.1 Seguridad
- **Contraseñas:** bcrypt
- **CSRF:** Token por sesión en todos los formularios POST
- **2FA:** PIN 6 dígitos vía SMTP, obligatorio por usuario
- **Rate limiting:** 3 intentos de login, 3 intentos de PIN, bloqueo 30 min
- **Historial inmodificable:** SHA-256, solo INSERT (sin DELETE/UPDATE)
- **Archivos:** Almacenados fuera de `public/`, servidos solo vía controlador
- **SQL Injection:** Prepared statements (PDO)
- **XSS:** `htmlspecialchars()` en todas las salidas
- **Directorio sensible:** Bloqueado por `.htaccess`: `basedatos|database|config|app|storage|vendor|docs`

### 4.2 Rendimiento
- **BD:** 29 tablas InnoDB con índices en FK y columnas de búsqueda
- **Notificaciones:** Pusher en tiempo real + polling 30s fallback
- **Paginación:** 20 registros por página en listados
- **Responsive:** Sidebar colapsable, tablas `table-responsive`, portal con cards apiladas en móvil
- **Carga bajo demanda:** Obligaciones se calculan al abrir sesión, no pre-calculadas

### 4.3 Mantenibilidad
- **Arquitectura MVC pura:** Sin framework, 23 controladores, 12 helpers, 3 modelos
- **Migraciones:** Automáticas en deploy (`database/migrate.php`)
- **Deploy:** Webhook GitHub → bash script (backup configs, git reset, migrations, composer)
- **Configuraciones producción:** Preservadas entre deploys (database.php, email.php, pusher.php)
- **AGENTS.md:** Contexto completo del proyecto para asistentes IA

### 4.4 Usabilidad
- Sidebar bifurcado según rol (admin completo / socio simplificado)
- Toggle light-dark en sidebar
- Modal de notificaciones unificado (reemplaza alert() nativos)
- Badges con contadores en tiempo real (Pusher)
- Summernote WYSIWYG para condiciones de productos
- Interface en español con acentos y ñ
